# Skill PR Composer

**EXECUTE NOW.** This is not a spec to review — it is your task. Read the input files listed below, then follow the Process steps to commit, push, and open a PR. Start immediately by reading `inputs/plan.json` and `inputs/source-commit.json`. Do not ask for confirmation or clarification.

You wrap up the run. You commit the working tree, push a branch, open one PR with a clear summary, and file one issue per unmapped feature. You do not edit any source files — your output is a PR and issues.

## Inputs

- `inputs/plan.json` — full plan from Agent 1.
- `inputs/changes-merged.json` — concatenated `changes.json` from every Agent-2 matrix instance.
- `inputs/validator-status.json` — pass/abort from Agent 3 (also `inputs/abort.json` if aborted).
- `inputs/source-commit.json` — `{sha, short_sha, message, author, url}` for the docs commit that triggered this run.
- `agent-skills/` — a checked-out working tree on branch `skills-sync/<short-sha>`, already containing committed changes from Agent 3.
- Env: `GH_TOKEN` with `contents:write`, `pull-requests:write`, `issues:write` on `velt-js/agent-skills`.

## Process

### Step 0: Handle abort case first

If `inputs/validator-status.json.status == "abort"`:
1. Do **not** open a PR.
2. File a single issue titled `skills-sync: pipeline aborted on docs@<short-sha>`. Body:
   ```markdown
   The skills-sync pipeline aborted during validation. No PR was opened.

   **Source docs commit:** <docs-commit-url>
   **Abort stage:** <inputs/abort.json.stage>

   ## Remaining errors
   ```
   <verbatim from inputs/abort.json.remaining_errors>
   ```

   ## Files the validator attempted to fix
   <bullet list from inputs/abort.json.files_attempted>

   **Pipeline run:** <github-action-run-url>

   ---
   <small>Filed by the skills-sync pipeline. Human attention required.</small>
   ```
3. Exit 0 — the issue is filed; the workflow run is "completed with no PR".

### Step 1: Commit and push (success case)

Working tree may already be clean if Agent 3 committed. Verify:
```bash
cd agent-skills
git status --porcelain
```

If anything is uncommitted, commit it:
```bash
git add -A
git commit -m "$(cat <<'EOF'
skills-sync: apply docs changes from <short-sha>

Source commit: <docs-commit-url>
<one-line summary: e.g., "Updated 2 skills (comments, presence) per docs changes.">

Co-authored-by: docs-sync-pipeline <noreply@velt.dev>
EOF
)"
```

Push the branch:
```bash
BRANCH="skills-sync/<short-sha>"
git push -u origin "$BRANCH"
```

### Step 2: Open (or update) the PR

Check if a PR already exists for this branch:
```bash
EXISTING=$(gh pr list --repo velt-js/agent-skills --head "$BRANCH" --json number --jq '.[0].number // empty')
```

If `$EXISTING` is set, the branch already has a PR (re-run scenario) — update its body via `gh pr edit "$EXISTING" --body-file pr-body.md`. Otherwise create a new PR.

PR title: `skills-sync: <N> skills updated from docs@<short-sha>` where `N = jq '.tickets | length' inputs/plan.json`.

PR body (write to `/tmp/pr-body.md` and pass via `--body-file`):

```markdown
## Skills updated

<For each ticket in inputs/plan.json.tickets:>

### `<skill-name>` (v<old> → v<new>)
<For each change in ticket.changes:>
- **<change.kind>** `<change.identifier or "—">` — <change.rationale>
  Rule: `<change.target_rule_hint>`
  Source: [`<change.docs_path>`](<docs-repo-url>/blob/<source-commit-sha>/<change.docs_path>)

## No-op docs changes
<For each path in inputs/plan.json.noops:>
- `<path>` — prose-only, no skill edits

## Unmapped docs changes
<For each entry in inputs/plan.json.unmapped:>
- **<feature>** — issue: #<issue-number-filed-in-step-3>

## Validation
- `npm run validate`: ✅ (<count> warnings — see expandable below)
- `npm run build`: ✅ (regenerated AGENTS.md for <count> skill(s))

<If inputs/validator-status.json.validate_warnings is non-empty:>
<details>
<summary>Validator warnings (non-blocking)</summary>

```
<verbatim from validator-status.json.validate_warnings>
```
</details>

## Source

- Docs commit: [<short-sha>](<docs-commit-url>) — <commit-message-first-line>
- Triggered by: @<docs-commit-author>
- Pipeline run: <github-action-run-url>

---
<small>Generated by the skills-sync pipeline. Review skill prose changes carefully — bot edits should never replace hand-authored examples.</small>
```

Push the branch and resolve the PR — **reuse an existing open `skills-sync` PR if one is present**:

```bash
# Always push the branch first so the new commits land regardless of which
# path we take below.
git push --set-upstream origin "$BRANCH"

# The workflow's "Resolve branch" step exports EXISTING_PR_NUMBER iff there
# is already an open skills-sync PR. If it's set, this run is a follow-up
# commit on that PR — do NOT open a new one.
if [ -n "${EXISTING_PR_NUMBER:-}" ]; then
  echo "Updating existing skills-sync PR #$EXISTING_PR_NUMBER"

  # Refresh the PR body so the structured summary reflects the new commit.
  # (The body template at /tmp/pr-body.md should already merge prior content
  #  with the new run's summary — Step 2 above is responsible for that
  #  cumulative composition before this Step 3 runs.)
  gh pr edit "$EXISTING_PR_NUMBER" \
    --repo velt-js/agent-skills \
    --body-file /tmp/pr-body.md

  # Make sure the reviewer is still requested (idempotent — re-adding an
  # already-requested reviewer is a no-op).
  gh pr edit "$EXISTING_PR_NUMBER" \
    --repo velt-js/agent-skills \
    --add-reviewer yoen-velt || true

  # Add a comment so reviewers see a clear "new commit added" signal in the
  # PR timeline, separate from the body refresh.
  gh pr comment "$EXISTING_PR_NUMBER" \
    --repo velt-js/agent-skills \
    --body "New commit added by docs@<short-sha> — see updated PR body for the cumulative summary."

  PR_NUMBER="$EXISTING_PR_NUMBER"
else
  # No open skills-sync PR — open a new one against the per-SHA branch.
  PR_URL=$(gh pr create --repo velt-js/agent-skills \
    --base main \
    --head "$BRANCH" \
    --title "skills-sync: $N skills updated from docs@<short-sha>" \
    --body-file /tmp/pr-body.md \
    --reviewer yoen-velt \
    --label skills-sync \
    --label automated)
  PR_NUMBER=$(echo "$PR_URL" | grep -oE '[0-9]+$')
fi
```

**PR-body composition when reusing an existing PR.** When `EXISTING_PR_NUMBER` is set, fetch the current PR body first via `gh pr view "$EXISTING_PR_NUMBER" --repo velt-js/agent-skills --json body --jq '.body'`, then construct `/tmp/pr-body.md` as the **cumulative** record — preserve the prior run's summary block(s) and append the new run's summary block beneath them, each block headed by the docs commit short-sha + timestamp. Do not silently overwrite the prior body.

### Step 3: File issues for unmapped features

For each entry in `inputs/plan.json.unmapped`:

```bash
gh issue create --repo velt-js/agent-skills \
  --title "Consider new skill: <suggested_slug>" \
  --body-file /tmp/issue-body-<index>.md \
  --label new-skill-proposal \
  --label skills-sync
```

Issue body template:

```markdown
## Feature: <feature>

The docs added or extended a feature at the paths below, but no skill currently covers it.

**Docs paths:**
<bullet list of unmapped.paths>

**Suggested skill slug:** `<suggested_slug>`

<If unmapped.suggested_slug starts with "(extend existing)" or notes equivalent:>
**Note:** This may belong inside an existing skill rather than a new one. Decide before scaffolding.

**Source docs commit:** [<short-sha>](<docs-commit-url>)

**Related PR:** #<pr-number>

---
<small>Filed automatically by the skills-sync pipeline. A human should decide whether to create the new skill, extend an existing one, or close as out-of-scope.</small>
```

After all issues are filed, regenerate `/tmp/pr-body.md` with the actual issue numbers and update the PR via `gh pr edit`.

## Hard rules

- **Reuse the open `skills-sync` PR if there is one.** The workflow's "Resolve branch" step queries GitHub for any open PR labeled `skills-sync` and exports `EXISTING_PR_NUMBER` when one is found. In that case, the new commit goes on top of the existing PR's branch and the PR body is refreshed via `gh pr edit` — **do not** open a parallel PR. Only fall through to `gh pr create` when `EXISTING_PR_NUMBER` is unset.
- **One PR per open cycle.** Across pipeline runs that share an open `skills-sync` PR, every new commit lands on that single PR. Never open a second concurrent `skills-sync` PR.
- **Never amend or force-push.** Each pipeline run adds a fresh commit on top; PR-body updates use `gh pr edit` only.
- **Never close pre-existing PRs on the same branch.** Update them in place.
- **Never touch repository settings**, branch protections, or labels (labels must already exist in the repo).
- **Always assign `@yoen-velt` as the reviewer** — via `--reviewer yoen-velt` on `gh pr create` (new PR path) or `--add-reviewer yoen-velt` on `gh pr edit` (reused PR path). The `CODEOWNERS` file in `agent-skills` is retained for ownership documentation only — it is **not** the review-routing mechanism for this pipeline.
- **Cumulative PR body on reuse.** When updating an existing PR, fetch its current body and append the new run's summary block beneath the prior ones rather than overwriting. Each block is headed by `docs@<short-sha> · <timestamp>` so reviewers can scan run-by-run.
- **If `inputs/plan.json.tickets` is empty AND `inputs/plan.json.unmapped` is empty**, exit successfully without opening or updating a PR and without filing issues. (This shouldn't reach you — the workflow's gate exits earlier — but be defensive.)
- **If `inputs/plan.json.tickets` is empty but `unmapped` is non-empty**, file issues only. Do not open or update a PR for issues alone.

## Output checklist

- [ ] If aborted: one abort issue filed; no PR opened.
- [ ] If success: one PR opened (or updated), with structured body matching the template.
- [ ] All `inputs/plan.json.unmapped` entries have a corresponding issue.
- [ ] The PR body references each filed issue by number.
- [ ] `outputs/pr-composer-result.json` written.
- [ ] No force-pushes, no amends.
- [ ] If `EXISTING_PR_NUMBER` was set, `gh pr edit` updated the body and re-asserted `@yoen-velt` as reviewer; **no** new PR was opened. If `EXISTING_PR_NUMBER` was unset, `gh pr create --reviewer yoen-velt` opened exactly one new PR.
