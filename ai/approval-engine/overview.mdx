## What is the Approval Engine?

The Approval Engine is a declarative workflow runtime for multi-step agent + human approval processes. Author a workflow definition once (nodes + edges + optional parallel groups), then dispatch it against a trigger to spawn a running execution. Every state change is persisted, replayable, and ‚Äî when you configure a webhook ‚Äî pushed to your receiver in real time.

You describe your approval flow as a DAG of `agent`, `human`, and `webhook` nodes connected by edges. The engine handles state, fan-out, parallel groups with quorum policies, SLA breaches, retries, idempotency, and event delivery. You record reviewer decisions and agent resolutions through the REST API; the engine advances the workflow and notifies your systems.

## Key capabilities

- **Declarative workflow definitions** ‚Äî Describe nodes (`agent` / `human` / `webhook`), edges with optional `when` expressions, and parallel groups with quorum policies. Definitions are versioned, with in-flight executions immune to updates.
- **Agent + human steps** ‚Äî Run AI agents non-blocking by default, or park them with `blocking: true` until an external resolution arrives. Human steps support multiple reviewers with mandatory/optional flags.
- **Parallel groups with quorum** ‚Äî Run reviewers in parallel and gate downstream work behind quorum policies: `waitAll`, `cancelOnQuorum`, or `joinOnQuorum`. Optionally require specific reviewers (`requiredNodeIds`) for quorum to count.
- **Idempotent dispatch** ‚Äî Replaying with the same `idempotencyKey` returns the original execution id rather than spawning duplicates.
- **Real-time webhooks** ‚Äî Every externally-visible state change is POSTed to your receiver with an HMAC-SHA256 signature and exponential-backoff retries. Missed events recoverable via `/executions/getEvents`.
- **SLA-aware** ‚Äî Set `slaMs` per step; the engine transitions to `breached` and routes via dedicated `breached` edges.
- **Static linting** ‚Äî Definitions are validated at create/update time for cycles, dangling edges, unreachable nodes, quorum misconfiguration, and group-membership conflicts.

## Key concepts

### Definition

A definition is the static blueprint of your approval workflow. It has:

- `definitionId` ‚Äî stable identifier used in every dispatch
- `name` ‚Äî human-readable label
- `nodes` ‚Äî work units that can run
- `edges` ‚Äî connections between nodes
- `groups` (optional) ‚Äî parallel-group declarations with quorum policies

Definitions are versioned. Every update increments the version and snapshots the prior content. In-flight executions are immune to updates.

**Node types:**

| Node type | Purpose | Parks in `waiting`? |
|-----------|---------|---------------------|
| `agent`   | Runs an agent (synthetic in v1; real-agent integration drops in without contract changes). Non-blocking by default. With `blocking: true`, the step parks until external resolutions arrive via `/steps/recordAgentResolution`. | Only when `blocking: true` |
| `human`   | Requires reviewer approval. Use `reviewers: [{ userId, mandatory }]` and drive via `/steps/recordReviewerDecision`. Legacy `reviewerIds[]` is still accepted for back-compat. | Yes |
| `webhook` | Schema-accepted for forward-compatibility; runtime handler is deferred in v1. | N/A |

Edges can carry an optional `when` expression (e.g. `output.passesBrandCheck == true`). Expressions are compiled at write time (pure AST, no eval) and walked at runtime. Supported operators: equality / comparison / boolean / regex / includes / startsWith / endsWith / length / isEmpty. Path roots: `output.*`, `step.*`, `execution.input.*`.

### Execution

An execution is one live run of a definition. Dispatch pins the definition version, stamps a correlation ID and idempotency key, and enqueues the first step(s).

| Field | Description |
|-------|-------------|
| `executionId`       | Server-generated. The handle for every subsequent operation. |
| `status`            | `pending ‚Üí running ‚Üí completed \| failed \| cancelled` |
| `correlationId`     | Threaded through every log line, event, and webhook for end-to-end tracing. |
| `idempotencyKey`    | Client-supplied or server-generated. Dispatching twice with the same key returns the same `executionId`. |
| `definitionVersion` | The pinned version. Updates to the definition after dispatch don't affect this execution. |
| `steps[]`           | One entry per step that has been scheduled or completed. |

### Step

A step is one runtime instance of a node. Status flow:

```
pending ‚Üí running ‚Üí (waiting) ‚Üí completed | failed | skipped | cancelled | breached
```

`waiting` only applies to human steps and blocking agent steps.

Step IDs are deterministic so retries land on the same doc:

- Root steps (no incoming edges): `step_<nodeId>_<timestamp>_<rand>`
- Per-edge fan-out: `${parentStepId}__to__${childNodeId}`
- `joinOnQuorum` group fan-out: `group_<groupId>__to__<childNodeId>` (single instance regardless of how many group members ran)

### Scope

| Level | Meaning |
|-------|---------|
| `apiKey`       | Workspace-wide. |
| `organization` | Bound to a specific `organizationId`. |
| `document`     | Bound to a specific `documentId` under an organization. |

Defaults to `{ level: "apiKey" }` if omitted on create.

### Idempotency

Dispatch is idempotent on `idempotencyKey`. Replaying with the same key returns `{ deduplicated: true, executionId: <original> }` rather than spawning a duplicate. Even a true contention race (two concurrent dispatches with the same key) returns the original execution id.

### Events and webhooks

Every state change writes an event doc with a monotonic `seq`. If you set `webhookUrl` + `webhookSecret` on dispatch, externally-visible event types are POSTed to your receiver with an HMAC-SHA256 signature and exponential-backoff retry. Missed events can be recovered via `/executions/getEvents` using `sinceSeq`.

## How it works

1. **Create a definition** ‚Äî Register a workflow blueprint with nodes (`agent`, `human`, `webhook`), edges that connect them, and optional parallel groups with quorum policies. The definition is linted at write time.
2. **Dispatch an execution** ‚Äî POST a `definitionId`, an idempotency key, and an optional `triggerContext` (exposed as `execution.input.*` in edge expressions). The engine pins the current definition version and enqueues the first step(s).
3. **Engine drives the workflow** ‚Äî Agent nodes run immediately. Human nodes (and blocking agent nodes) park in `waiting` until you record decisions or resolutions through the REST API.
4. **Record decisions** ‚Äî Call `/steps/recordReviewerDecision` for human steps and `/steps/recordAgentResolution` for blocking agent steps. Edges with matching `when` expressions fire; quorum policies trigger their configured side effects.
5. **Consume events** ‚Äî Subscribe via webhook for real-time delivery, or poll `/executions/getEvents` with `sinceSeq` to reconcile after an outage.
