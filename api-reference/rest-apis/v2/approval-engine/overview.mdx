The Approval Engine is a declarative workflow runtime for multi-step agent + human approval processes. Author a workflow definition once (nodes + edges + optional parallel groups), then dispatch it against a trigger to spawn a running execution. Every state change is persisted, replayable, and — when you configure a webhook — pushed to your receiver in real time.

## Base URL

| Environment | Base URL |
|-------------|----------|
| Production  | `https://api.velt.dev` |
| Staging     | `https://staging.velt.dev` |

All endpoints are versioned. `/v1/...` and `/v2/...` are active aliases that accept the same request shape and return the same response shape. New integrations should prefer V2.

## Core Concepts

### Definition

A definition is the static blueprint of your approval workflow. It has:
- `definitionId` — stable identifier used in every dispatch
- `name` — human-readable label
- `nodes` — work units that can run
- `edges` — connections between nodes
- `groups` (optional) — parallel-group declarations with quorum policies

Definitions are versioned. Every update increments the version and snapshots the prior content. In-flight executions are immune to updates.

**Node types:**

| Node type | Purpose | Parks in `waiting`? |
|-----------|---------|---------------------|
| `agent`   | Runs an agent (synthetic in v1; real-agent integration drops in without contract changes). Non-blocking by default. With `blocking: true`, the step parks until external resolutions arrive via `/steps/recordAgentResolution`. | Only when `blocking: true` |
| `human`   | Requires reviewer approval. Use `reviewers: [{ userId, mandatory }]` and drive via `/steps/recordReviewerDecision`. Legacy `reviewerIds[]` is still accepted for back-compat. | Yes |
| `webhook` | Schema-accepted for forward-compatibility; runtime handler is deferred in v1. | N/A |

Edges can carry an optional `when` expression (e.g. `output.passesBrandCheck == true`). Expressions are compiled at write time (pure AST, no eval) and walked at runtime. Supported operators: equality / comparison / boolean / regex / includes / startsWith / endsWith / length / isEmpty. Path roots: `output.*`, `step.*`, `execution.input.*`.

### Execution

An execution is one live run of a definition. Dispatch pins the definition version, stamps a correlation ID and idempotency key, and enqueues the first step(s).

| Field | Description |
|-------|-------------|
| `executionId`       | Server-generated. The handle for every subsequent operation. |
| `status`            | `pending → running → completed \| failed \| cancelled` |
| `correlationId`     | Threaded through every log line, event, and webhook for end-to-end tracing. |
| `idempotencyKey`    | Client-supplied or server-generated. Dispatching twice with the same key returns the same `executionId`. |
| `definitionVersion` | The pinned version. Updates to the definition after dispatch don't affect this execution. |
| `steps[]`           | One entry per step that has been scheduled or completed. |

### Step

A step is one runtime instance of a node. Status flow:

```
pending → running → (waiting) → completed | failed | skipped | cancelled | breached
```

`waiting` only applies to human steps and blocking agent steps.

Step IDs are deterministic so retries land on the same doc:
- Root steps (no incoming edges): `step_<nodeId>_<timestamp>_<rand>`
- Per-edge fan-out: `${parentStepId}__to__${childNodeId}`
- `joinOnQuorum` group fan-out: `group_<groupId>__to__<childNodeId>` (single instance regardless of how many group members ran)

### Scope

| Level | Meaning |
|-------|---------|
| `apiKey`       | Workspace-wide. |
| `organization` | Bound to a specific `organizationId`. |
| `document`     | Bound to a specific `documentId` under an organization. |

Defaults to `{ level: "apiKey" }` if omitted on create.

### Idempotency

Dispatch is idempotent on `idempotencyKey`. Replaying with the same key returns `{ deduplicated: true, executionId: <original> }` rather than spawning a duplicate. Even a true contention race (two concurrent dispatches with the same key) returns the original execution id.

### Events and Webhooks

Every state change writes an event doc with a monotonic `seq`. If you set `webhookUrl` + `webhookSecret` on dispatch, externally-visible event types are POSTed to your receiver with an HMAC-SHA256 signature and exponential-backoff retry. Missed events can be recovered via `/executions/getEvents` using `sinceSeq`.

## Parallel Groups and Quorum Policies

A parallel group declares a set of member nodes that conceptually run in parallel and share an approval threshold.

```json
{
  "groupId": "parallel-review",
  "memberNodeIds": ["human-legal", "human-brand"],
  "expectedSteps": 2,
  "quorum": 2,
  "onQuorumMet": "waitAll"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `groupId`         | string   | yes | 1–64 chars. Stable identifier. |
| `memberNodeIds`   | string[] | yes | 1–500 nodes. Each must be declared as a top-level node. A node may belong to at most one group. |
| `expectedSteps`   | integer  | yes | 1–500. Total members the group expects to terminate. Must equal `memberNodeIds.length`. |
| `quorum`          | integer  | yes | 1–`expectedSteps`. The number of approvals required to fire the policy's side effect. |
| `onQuorumMet`     | enum     | no  | `waitAll` (default) / `cancelOnQuorum` / `joinOnQuorum`. |
| `requiredNodeIds` | string[] | no  | Specific members whose approval is required for quorum to be met. Every entry must also be in `memberNodeIds`. `length` must be `<= quorum`. |

### Quorum Is Approval Count, Not Completion Count

A member counts as an approval when its terminal status is `completed` AND its `output.decision === 'approve'`. Rejections, failures, breaches, and cancellations contribute to the completion counter only — never the approval counter.

Two consequences:
1. Non-blocking agent members never satisfy quorum — they have no decision concept. Only place human or blocking agent nodes inside groups whose policy is `cancelOnQuorum` or `joinOnQuorum`.
2. A `reject` does not block the group from rolling up to "complete" — it just keeps `approvedShards` from advancing. Group-completion (`expectedSteps` met) and group-quorum (approval threshold) are tracked separately.

### onQuorumMet Policies

| Policy | Side effect on first-time approval-quorum-met | Per-member fan-out |
|--------|------------------------------------------------|--------------------|
| `waitAll` (default) | Emits `group.quorum-met` event only. Group is purely informational. | Each member's outgoing edges fire on its own completion. If two members both fan out to the same downstream node, you get two downstream step instances. |
| `cancelOnQuorum`    | Emits `group.quorum-met` AND cancels every sibling member step still in `waiting` (system-actor cancellation, audit reason `group-quorum-met`). | Each completing member still fans out per-edge. Cancelled siblings do not fan out. |
| `joinOnQuorum`      | Emits `group.quorum-met`, cancels waiting siblings, AND fires a single group-owned downstream fan-out: one new step per shared outgoing-edge target with deterministic stepId `group_<groupId>__to__<childNodeId>`. The successor's input is `{ groupOutputs, groupId, quorum, totalApproved }`. | Suppressed for group members — the group container owns fan-out, so downstream successors run exactly once. |

### Specific-Must-Approve Quorum

By default, quorum is anonymous: any N approvals out of M members trigger the policy. To express "these specific members must approve", declare them in `requiredNodeIds`:

```json
{
  "groupId": "approver-group",
  "memberNodeIds": ["legal", "finance", "brand"],
  "expectedSteps": 3,
  "quorum": 2,
  "requiredNodeIds": ["legal", "finance"]
}
```

Quorum-met now requires both:
1. Every `nodeId` in `requiredNodeIds` is among the approvers, AND
2. Total approval count reaches the numeric `quorum`.

In the example, `brand` alone approving doesn't satisfy quorum even if `quorum: 2` is reached numerically — `legal` AND `finance` must both also approve. If `requiredNodeIds` is omitted or empty, behavior collapses back to anonymous quorum.

## Linter Rules

Definitions are linted at create/update time. Any rule violation is rejected with `INVALID_ARGUMENT` and an explicit code in the error message.

| Code | Meaning |
|------|---------|
| `duplicate-node-id` | Two nodes share the same `nodeId`. |
| `dangling-edge` | Edge references a `from` or `to` that isn't declared. |
| `cycle-detected` | The graph contains a cycle. v1 is DAG-only. |
| `unreachable-node` | A node has no path from any root. |
| `node-missing-config` | A node has no config block. |
| `missing-breach-edge` | A node has `slaMs` set but no outgoing edge that routes on `status == 'breached'` — breaches would silently dead-end. |
| `group-duplicate-id` | Two groups share the same `groupId`. |
| `group-members-empty` | `memberNodeIds` is empty. |
| `group-member-missing` | A member references an unknown node. |
| `group-expected-steps-invalid` | `expectedSteps < 1`. |
| `group-quorum-invalid` | `quorum < 1` or `quorum > expectedSteps`. |
| `group-cancelonquorum-requires-quorum-lt-expected` | `cancelOnQuorum` requires `quorum < expectedSteps`. |
| `group-joinonquorum-members-must-share-successors` | `joinOnQuorum` requires every member to have an identical set of outgoing-edge target nodes. |
| `group-required-not-in-members` | An entry in `requiredNodeIds` is not in `memberNodeIds`. |
| `group-required-exceeds-quorum` | `requiredNodeIds.length > quorum`. |
| `group-node-in-multiple-groups` | A node appears as a member of two or more groups. |

Each node also accepts an optional top-level field:
- `slaMs` (integer, ms) — SLA deadline for the step. If the step doesn't complete within this window, it transitions to `breached`. See the `missing-breach-edge` [linter rule](#linter-rules).

**Node config by type:**
- **`agent`**: `agentId` (required), `promptOverride` (≤ 8000 chars), `inputMapping`, `blocking` (default `false`), `resolutionPolicy` (required when `blocking: true`): `{ kind: "allResolved" | "minResolved", minCount?: integer }` — `minCount` is required when `kind === "minResolved"`. `agentMaxRuntimeMs` (≤ 86400000). `requireNonEmptyOutput` (boolean).
- **`human`**: exactly one of `reviewers[]` (preferred: `[{ userId, mandatory }]`) or `reviewerIds[]` (legacy). `reviewers[]` must include at least one `mandatory: true`. UserIds must be unique. Optional `commentBody` (≤ 8000 chars).
- **`webhook`**: Runtime handler is deferred in v1.

**Node object example:**

```json
{
  "nodeId": "brand-check",
  "type": "agent",
  "config": { "agentId": "brand-agent-v1", "blocking": false, "requireNonEmptyOutput": true },
  "slaMs": 3600000
}
```

**Edge object example:**

```json
{ "from": "brand-check", "to": "legal-review", "when": "output.passesBrandCheck == true" }
```

`when` is an edge-gating expression evaluated against the source step's output. Optional — if omitted, the edge always fires.

#### Record Reviewer Decision

`POST /v2/workflow/steps/recordReviewerDecision`

Records a single reviewer's decision against a human step. When all mandatory reviewers approve (or any reviewer rejects), the step's aggregator transitions terminal and the step resumes.

<ParamField body="data" type="object" required>
  <Expandable title="properties">
    <ParamField body="executionId" type="string" required>

      Must reference a human step in `waiting`.
    </ParamField>
    <ParamField body="reviewerId" type="string" required>
      Must match a `userId` declared on the node's `reviewers[]`.
    </ParamField>
    <ParamField body="decision" type="string" required>
      `approve` / `reject`.
    </ParamField>
    <ParamField body="reason" type="string">
      ≤ 2000 chars.
    </ParamField>
  </Expandable>

## Webhook Push

When an execution is dispatched with `webhookUrl` + `webhookSecret`, every externally-visible event is POSTed to your receiver.

**Delivery:** POST, JSON body, 10s timeout, no redirects.

**Headers:**

| Header | Description |
|--------|-------------|
| `x-velt-signature` | `sha256=<hex>` — HMAC-SHA256 of the raw request body. |
| `x-velt-event-id` | Stable event ID, unchanged across retries. |
| `x-velt-attempt` | 0-based attempt counter. |

**Payload example:**

```json
{
  "event": "step.awaiting-approval",
  "executionId": "exec_...",
  "definitionId": "marketing-copy-approval",
  "stepId": "step_agent-draft_..._lwofay__to__human-legal",
  "status": "waiting",
  "seq": 7,
  "timestamp": "2026-04-28T08:05:00.000Z",
  "correlationId": "corr_campaign_42",
  "data": { "waitingForReviewers": 1, "mandatoryCount": 1, "resumeKey": "human:..." }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `event`         | string             | External event type. See [Event Reference](#event-reference). |
| `executionId`   | string             | The execution this event belongs to. |
| `definitionId`  | string             | The pinned definition id for this execution. |
| `stepId`        | string \| null     | Set for step-level events; `null` for execution-level. |
| `status`        | string \| undefined | Optional. Extracted from `data.status` when present. |
| `seq`           | integer            | Monotonic per-execution sequence. Use for ordering and reconciliation. |
| `timestamp`     | string             | ISO 8601 timestamp of the event. |
| `correlationId` | string             | End-to-end trace id (echoed from dispatch). |
| `data`          | object             | Event-specific payload. |

### Verifying the Signature (Node.js)

```javascript
const crypto = require('crypto');

function verifyVeltSignature(rawBody, headerValue, secret) {
  const [scheme, hex] = String(headerValue).split('=');
  if (scheme !== 'sha256' || !hex) return false;
  const computed = crypto
    .createHmac('sha256', secret)
    .update(rawBody, 'utf8')
    .digest('hex');
  const a = Buffer.from(hex, 'hex');
  const b = Buffer.from(computed, 'hex');
  return a.length === b.length && crypto.timingSafeEqual(a, b);
}
```

<Note>
  Hash the raw request body bytes — do not re-serialize the parsed JSON object.
</Note>

### Retry Policy

| Attempt | Delay before retry |
|---------|--------------------|
| 1 (initial) | — |
| 2 | 2 s |
| 3 | 8 s |
| 4 | 32 s |
| 5 | 2 min |
| 6 | 8 min → DLQ |

After 5 failed retries, the payload is written to a dead-letter queue. Recover missed events via `/executions/getEvents` with `sinceSeq`.

**At-least-once delivery:** the same `eventId` + `seq` appear on retries — make your receiver idempotent on `(executionId, seq)`.

## Event Reference

Externally-visible events delivered via webhook and returned from `/executions/getEvents`:

| Event type (external) | Internal name | When emitted | `data` highlights |
|-----------------------|---------------|--------------|-------------------|
| `execution.dispatched` | same | Execution created; first step(s) scheduled. | `{ definitionId, definitionVersion, rootStepIds }` |
| `execution.completed`  | same | All steps terminal, no unhandled failures. | null |
| `execution.failed`     | same | Any blocking step ended in `failed` or `breached` without a recovery edge. | `{ failureReason }` |
| `execution.cancelled`  | same | `/executions/cancel` or full-execution rollback. | `{ reason? }` |
| `step.awaiting-approval` | `step.waiting` | A human or blocking-agent step entered `waiting`. | `{ waitingForReviewers, mandatoryCount, resumeKey }` |
| `step.completed`       | same | Step transitioned to `completed`. | For human/blocking-agent: `{ aggregatorStatus, nodeType, decision, aggregatorBacked }`. For non-blocking agents: `{ agentId }`. |
| `step.failed`          | same | Step transitioned to `failed` (retry budget exhausted). | `{ error: { code, message } }` |
| `step.breached`        | same | Step exceeded its configured SLA before completing. | `{ reason }` |
| `step.cancelled`       | same | Step cancelled via `/steps/cancel` or by quorum-met side effect. | `{ actorId, reason }` |
| `group.quorum-met`     | `parallel-group.quorum-met` | A parallel group's approval threshold was first satisfied. | `{ groupId, total, quorum, completedTotal, expectedSteps }` |

  Internal-only events (`step.scheduled`, `step.started`, `step.retried`, `step.resumed`, `step.response-recorded`, `step.overridden`, `parallel-group.completed`, `idempotency.suppressed`) fill `seq` gaps but are filtered from external delivery. Your stream may have non-contiguous `seq` values.
</Note>

## Error Reference

All errors follow the standard envelope:

```json
{ "error": { "message": "...", "status": "INVALID_ARGUMENT", "details": {} } }
```

### Canonical Codes

| Code | Meaning | Typical cause |
|------|---------|---------------|
| `INVALID_ARGUMENT`     | Schema or linter failure. | Missing field, wrong type, value out of range, linter rule violation. |
| `UNAUTHENTICATED`      | Missing or invalid `x-velt-auth-token`. | |
| `PERMISSION_DENIED`    | Auth token valid but lacks the required scope. | Non-admin attempting `/steps/cancel` or `/steps/resolve`. |
| `NOT_FOUND`            | Target doc does not exist. | Unknown `executionId`, `definitionId`, or `stepId`. |
| `ALREADY_EXISTS`       | Conflicting create. | Creating a definition with a `definitionId` already in use. |
| `FAILED_PRECONDITION`  | Optimistic lock or state-machine violation. | `ifVersion` mismatch on update; cancelling a terminal step; deleting a definition with in-flight executions. |
| `RESOURCE_EXHAUSTED`   | Rate limit exceeded. | Per-IP or per-API-key quota. |
| `DEADLINE_EXCEEDED`    | Internal timeout. | Retry with idempotency. |

### Schema-Level Validation Errors

| `message` | Trigger |
|-----------|---------|
| `webhookUrl and webhookSecret must be provided together` | Dispatch supplied one but not the other. |
| `webhookUrl must use https scheme` | Non-HTTPS scheme. |
| `webhookUrl host resolves to a private, loopback, or link-local address` | Literal private IP, localhost, metadata.google.internal, or *.internal. |
| `at least one of reviewerIds or reviewers must be provided` | Human node with no reviewers. |
| `cannot set both reviewerIds and reviewers — use one` | Both populated — pick the modern `reviewers[]` form. |
| `reviewer userIds must be unique` | Duplicate userId in `reviewers[]`. |
| `reviewers must include at least one mandatory reviewer (allMandatoryApproved would otherwise never resolve)` | Every `reviewer.mandatory === false`. |
| `resolutionPolicy required when blocking === true` | Blocking agent node without a policy. |
| `minCount required when kind === "minResolved"` | `resolutionPolicy.kind = "minResolved"` with no `minCount`. |

## Versioning and Stability

- `/v1/...` and `/v2/...` are active aliases for the same controllers. New integrations should use V2.
- The event surface, webhook payload shape, and error envelope are stable for v1.
- Internal-only event types may change without notice; do not build on them.
- Definition updates always produce a new version — in-flight executions never change shape under you.
- The `data.reason` field on `step.cancelled` events is an open string set; new reasons may be added without a breaking change.

## Rate Limiting

Rate limits are applied per IP + API key, with additional per-endpoint tiers on high-volume routes. A `RESOURCE_EXHAUSTED` error indicates you should back off with exponential retry. Dispatch retries are safe to replay with an `idempotencyKey`.

## What's Deferred for v1.x

- **Loop-back / kick-back-for-revision.** v1 is DAG-only; cycles are rejected by the linter. Workaround: dispatch a fresh execution treating revision as a new run.
- **Dynamic reviewer assignment.** Reviewers must be statically declared per node.
- **Conditional group membership.** Group `memberNodeIds` is static. Workaround: skip a reviewer node via `when` expressions on edges.
- **Per-reviewer SLA.** `slaMs` is node-level, not reviewer-level.
- **Webhook node runtime.** `type: "webhook"` is schema-accepted but the runtime handler is deferred. Webhook nodes resolve as `step.failed` with code `webhook-node-deferred`.
