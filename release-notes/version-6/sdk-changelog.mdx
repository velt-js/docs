<Update label="6.0.0-beta.3" description="June 30, 2026">

### New Features

- [**Comments**]: Added a `restrictTextSearchToAnchor` / `restrict-text-search-to-anchor` prop on `<VeltComments>` / `<velt-comments>` (default: `false`) and `enableRestrictTextSearchToAnchor()` / `disableRestrictTextSearchToAnchor()` API methods that control how the SDK re-locates a text comment's highlight. By default the SDK falls back to a wider (body-level or location-scoped) search when the commented text is no longer found inside the comment's original anchor element; set it to `true` to restrict the search to the anchor element only, with no fallback, so a comment ghosts immediately if its anchor text is gone. [Learn more →](/async-collaboration/comments/customize-behavior)

<Tabs>
<Tab title="React / Next.js">
```tsx
import { VeltComments } from '@veltdev/react';

// Using prop
<VeltComments restrictTextSearchToAnchor={true} />

// Using API methods
const commentElement = client.getCommentElement();

// Restrict the highlight search to the anchor element only (no body-level fallback).
commentElement.enableRestrictTextSearchToAnchor();

// Allow the body-level fallback search before ghosting (default behavior).
commentElement.disableRestrictTextSearchToAnchor();
```
</Tab>
<Tab title="Other Frameworks">
```html
<!-- Using attribute -->
<velt-comments restrict-text-search-to-anchor="true"></velt-comments>
```
```js
// Using API methods
const commentElement = Velt.getCommentElement();

// Restrict the highlight search to the anchor element only (no body-level fallback).
commentElement.enableRestrictTextSearchToAnchor();

// Allow the body-level fallback search before ghosting (default behavior).
commentElement.disableRestrictTextSearchToAnchor();
```
</Tab>

### Improvements

- [**Self-Hosting**]: The `fieldsToRemove` allowlist now applies to reaction annotations, recorder annotations, and all activity feature types (previously only comments and `custom` activities). List reaction- and recorder-specific custom fields in your resolver's `fieldsToRemove` config to store those values only in your own backend; they round-trip transparently on read via the resolver cache. In these newly supported reaction, recorder, and activity paths, fields are matched on `!== undefined`, so falsy-but-meaningful values (`0`, `false`, `""`) are moved too. The existing comment path remains truthy-gated. [Learn more →](/self-host-data/overview#excluding--extending-fields)

```ts
// Fields listed here are written to your storage (stripped data) and deleted
// from the payload sent to Velt's database; they are restored on read.
reactionAnnotationDataProvider = {
  config: {
    fieldsToRemove: ['internalRef', 'tenantId'],
  },
  // ...resolver implementation
};
```

- [**Comments / Sidebar V2**]: In the V2 comment sidebar, explicitly selecting a status in the status field filter now overrides the "hide resolved by default" rule. The `all`, `unread`, `read`, `open`, and `assignedToMe` presets still hide terminal/resolved comments by default, but picking "Resolved" (or any terminal status) now surfaces those comments instead of returning an empty or short list. [Learn more →](/async-collaboration/comments-sidebar/v2/customize-behavior)

### Bug Fixes

- [**Comments**]: When a text comment's stored text can no longer be found inside its resolved anchor element, the SDK now retries the search in a wider scope before ghosting the comment — `document.body` for body-anchored comments and the comment's location element for location-scoped comments (so the fallback never escapes the location). Text comments whose anchor went stale now re-highlight at their recovered location instead of silently ghosting; customers who require strict anchor-only matching can opt out with `restrictTextSearchToAnchor` (see New Features). [Learn more →](/async-collaboration/comments/customize-behavior)

- [**Comments / Sidebar V2**]: Per-field facet option counts are now computed against the same boundaries as the visible list (page mode plus the client/system filters), so every facet count can be realized by selecting that option. Previously a facet could show a non-zero count that filtered the list to nothing once selected. [Learn more →](/async-collaboration/comments-sidebar/v2/customize-behavior)

- [**Types**]: `@veltdev/types` is a declarations-only package, but its published entry chain re-exported symbols with plain `export *`, turning every value-bearing export (53 enums, 19 element-model classes, 2 client-model classes) into a phantom value that type-checked but resolved to `undefined` at runtime. The entire published entry chain is now type-only, so a consumer can no longer accidentally import a type-only symbol as a runtime value.

- [**Types**]: Published element-model declarations are now re-synced with the runtime SDK, and `// @ts-nocheck` has been removed from the entry/model declarations. The corrected public surface includes:
  - Methods that do not return a value now use `void`, including Notification history panel methods such as `openHistoryPanel()`, `closeHistoryPanel()`, and `toggleHistoryPanel()`.
  - `RecorderElement.initRecording(type, panelId?)` now returns `Promise<void>`.
  - Tag declarations use `TagAnnotation`; the stale `TagPinAnnotation` alias is removed, and `getAllTagAnnotations()` returns `Observable<TagAnnotation[] | null>`.
  - Presence declarations include `enableFlockMode(options?)`, `disableFlockMode()`, `startFollowingUser(userId, name)`, and `stopFollowingUser()`. The stale `enableFollowAlongMode()` and `disableFollowAlongMode()` declarations are removed.
  - Notification declarations include `enableCrossOrganization(config?)`, `disableCrossOrganization()`, and typed `on<T>(action)` subscriptions.
  - Rewriter declarations include `on<T>(action)`, `askAi(request)`, `replaceText(request)`, and `addComment(request)`.
  - Recorder declarations include `enableDarkMode()` and `disableDarkMode()`.
  - `Snippyly.identify()` now accepts [`Options`](/api-reference/sdk/models/data-models#options-2) instead of the non-existent `UserOptions`.

</Update>

<Update label="6.0.0-beta.1" description="June 23, 2026">

First beta of the modular SDK. The public `Velt` browser API is fully backward compatible — existing browser integrations work unchanged.

### New Features

- [**Core**]: **Modular SDK** — each feature now loads as its own on-demand chunk. Pass `featureAllowList` at init (e.g. `['comment', 'presence']`) to preload only the features you use; omit it to keep the existing behavior of preloading everything. [Learn more →](/api-reference/sdk/models/data-models)

- [**Core**]: New `preload<Feature>()` methods (e.g. `preloadComment()`) to warm a feature chunk ahead of use, or to load tag-only features (`userInvite`, `userRequest`, `videoPlayer`). [Learn more →](/api-reference/sdk/api/api-methods)

</Update>
